Python 源码讲解（复现同时学原理）

下面这部分不是“怎么敲命令”，而是“为什么命令背后的 Python 要这样写”。建议打开源码边看边对照。

A) 训练导出脚本：pc_train/train_and_export.py

调用链（点开看全局结构） 
main()
 ├─ parse_args()              # 解析 data/model/imgsz/epochs/opset 等参数
 ├─ train_if_needed(args)     # 训练或直接读取已有 best.pt
 │   └─ resolve_best_pt(...)  # 自动定位 best.pt，避免路径差异导致失败
 └─ export_model.export(...)  # 导出 ONNX，固定 opset=13
重点1：为什么要单独做 resolve_best_pt？

Ultralytics 在不同版本、不同 project/name 下，权重输出路径可能变化。脚本通过多级回退寻找 `best.pt`，把“手工找路径”变成自动化，工程更稳定。

重点2：为什么强调 opset=13？

这是你当前 Ascend+CANN 环境下的兼容优先值。opset 过高时，ATC 常出现算子不支持或解析失败。工程上优先“稳定可编译”。

B) 板端推理脚本：board_deploy/infer_image.py

从图片到结果图的完整数据流 
main()
 ├─ preprocess(image, imgsz)
 │   ├─ resize 到 640x640
 │   ├─ BGR -> RGB
 │   ├─ float32 + 归一化到 [0,1]
 │   └─ HWC -> CHW -> NCHW
 ├─ session = InferSession(device_id, om_path)
 ├─ pred_raw = session.infer([inp], mode="static")[0]
 ├─ pred = decode_output(pred_raw)      # 统一输出形状
 ├─ detections = postprocess(...)       # 阈值筛选 + NMS
 ├─ draw_boxes(...)
 └─ session.free_resource()             # 释放 ACL/NPU 资源
重点3：为什么输入必须转成 NCHW？

OpenCV 读图默认是 HWC，而模型输入通常是 NCHW。若不转置，通道顺序会错，推理结果会明显异常或完全不可用。

重点4：为什么输出形状会是 (1,84,8400) 或 (1,8400,84)？

84 = 4个框参数 + 80个类别分数；8400 是候选位置总数。`decode_output` 做的事就是把不同布局统一成同一种后处理格式。

重点5：NMS 在干什么？

模型会给同一目标吐出多个重叠框。NMS 会在重叠框里保留高分框，抑制低分重复框，让结果更干净。

重点6：为什么一定要 session.free_resource()？

Ascend 设备上下文和显存资源不是 Python 垃圾回收能及时处理的。显式释放可避免下次运行出现设备占用或初始化失败。

常见错误与处理（本项目实战版）

`ModuleNotFoundError: No module named 'ultralytics'`（PC）

原因：没有在 `pc_train` 虚拟环境安装依赖。执行步骤 1 后重试。

`source: command not found`（PC）

原因：把 Linux 命令放到 Windows PowerShell 执行了。`source` 只能在板端 Bash 执行。

`No module named 'tbe'`（Board ATC）

先执行：`source /usr/local/Ascend/ascend-toolkit/set_env.sh && python3 -c "import tbe"`。若失败则检查 CANN 安装与 Python 环境。

`ais_bench: command not found`（Board）

可以使用 `python3 -m ais_bench`，或者直接用本项目 `run_infer.sh` + `infer_image.py` 路线。

输入密码后“没反应”

ATC 编译可能耗时几十秒到几分钟。开第二个 SSH 窗口检查：`ps -ef | grep atc | grep -v grep`。
