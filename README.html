Python 源码讲解（复现同时学原理）

下面这部分不是“怎么敲命令”，而是“为什么命令背后的 Python 要这样写”。建议打开源码边看边对照。

A) 训练导出脚本：pc_train/train_and_export.py

调用链（点开看全局结构） 
main()
 ├─ parse_args()              # 解析 data/model/imgsz/epochs/opset 等参数
 ├─ train_if_needed(args)     # 训练或直接读取已有 best.pt
 │   └─ resolve_best_pt(...)  # 自动定位 best.pt，避免路径差异导致失败
 └─ export_model.export(...)  # 导出 ONNX，固定 opset=13
重点1：为什么要单独做 resolve_best_pt？

Ultralytics 在不同版本、不同 project/name 下，权重输出路径可能变化。脚本通过多级回退寻找 `best.pt`，把“手工找路径”变成自动化，工程更稳定。

重点2：为什么强调 opset=13？

这是你当前 Ascend+CANN 环境下的兼容优先值。opset 过高时，ATC 常出现算子不支持或解析失败。工程上优先“稳定可编译”。

B) 板端推理脚本：board_deploy/infer_image.py

从图片到结果图的完整数据流 
main()
 ├─ preprocess(image, imgsz)
 │   ├─ resize 到 640x640
 │   ├─ BGR -> RGB
 │   ├─ float32 + 归一化到 [0,1]
 │   └─ HWC -> CHW -> NCHW
 ├─ session = InferSession(device_id, om_path)
 ├─ pred_raw = session.infer([inp], mode="static")[0]
 ├─ pred = decode_output(pred_raw)      # 统一输出形状
 ├─ detections = postprocess(...)       # 阈值筛选 + NMS
 ├─ draw_boxes(...)
 └─ session.free_resource()             # 释放 ACL/NPU 资源
重点3：为什么输入必须转成 NCHW？

OpenCV 读图默认是 HWC，而模型输入通常是 NCHW。若不转置，通道顺序会错，推理结果会明显异常或完全不可用。

重点4：为什么输出形状会是 (1,84,8400) 或 (1,8400,84)？

84 = 4个框参数 + 80个类别分数；8400 是候选位置总数。`decode_output` 做的事就是把不同布局统一成同一种后处理格式。

重点5：NMS 在干什么？

模型会给同一目标吐出多个重叠框。NMS 会在重叠框里保留高分框，抑制低分重复框，让结果更干净。

重点6：为什么一定要 session.free_resource()？

Ascend 设备上下文和显存资源不是 Python 垃圾回收能及时处理的。显式释放可避免下次运行出现设备占用或初始化失败。

“模型源码”到底指什么？`yolov8n.pt` 不是代码文件啊

yolov8n.pt 是模型参数（权重），不是网络结构源码。网络结构和训练逻辑在 Ultralytics Python 包里。

你可以这样理解：

1) 代码（Python 类）定义模型怎么前向、怎么训练；2) `.pt` 是这些代码训练出来的参数快照；3) 运行时是“代码 + 参数”共同决定行为。

从 COCO8 到你自己的数据集，源码层需要改什么？

绝大多数情况下不改 Ultralytics 源码，只改 YAML：

1) 改 path/train/val 指向你的目录；2) 改 names 为你的类别；3) 标签格式遵循 YOLO txt（class x_center y_center w h，归一化）。

只有在你要改增强策略、采样逻辑、损失定义时，才需要改 `DetectionTrainer` / `YOLODataset` 相关源码。

为什么你这个项目里“源码学习”是值得的？

因为你已经把工程链路跑通了。下一阶段优化效果（漏检、误检、速度）时，瓶颈就在这两层：

1) 数据层（COCO8 或自定义数据集质量）；2) 训练/推理参数层（阈值、输入尺寸、数据增强、类别过滤）。

看懂源码调用链后，你能判断“该调参数”还是“该改代码”。

E) 工程编排层源码讲解（为什么一条命令能跑全流程）

scripts/run_end2end.ps1 的控制流是什么？ 
run_end2end.ps1
 ├─ if not SkipExport -> pc_export_onnx.ps1
 ├─ pc_upload_onnx.ps1
 ├─ pc_sync_board_scripts.ps1
 ├─ pc_remote_convert.ps1
 ├─ pc_remote_infer.ps1
 └─ pc_pull_result.ps1

本质上它是一个“串行状态机”。每一步成功才会进入下一步，任一步失败立即抛错终止，避免错误被后续步骤掩盖。

为什么要有 project_config.json？

它是单一事实来源（Single Source of Truth）：IP、用户、SOC、路径、阈值都集中管理。脚本只读配置，不在代码里硬编码环境信息。

F) 远程执行层原理（PC 调 Board 时到底发生了什么）

pc_remote_convert.ps1 / pc_remote_infer.ps1 的核心机制

1) 先在 PC 组装远程命令字符串；2) 用 ssh 发给板端；3) 板端 Bash 再解释执行；4) 退出码回传给 PC 脚本判断成功/失败。

这就是为什么“命令拼接和引号边界”很关键：错误的引号会把参数拆坏，导致看起来像是板端环境问题。

为什么要专门有 pc_sync_board_scripts.ps1？

它保证板端脚本版本与 PC 项目一致（目录创建 + scp 同步 + chmod）。这一步解决了“板端没有最新脚本/路径不存在”的工程一致性问题。

G) 编译适配层原理（ATC 不只是“一条命令”）

board_convert/convert_to_om.sh 做了哪些防御性设计？

1) 参数校验（路径、输入名、输入形状）；2) 自动处理 .om 后缀；3) source set_env.sh 载入 CANN；4) import tbe 预检查；5) 失败时给出明确排查提示。

为什么有 set +u / set -u？

部分系统下 set_env.sh 会引用未定义变量（如 LD_LIBRARY_PATH）。在严格模式 set -u 下会直接退出。临时关闭是为兼容现实环境差异。

常见错误与处理（本项目实战版）

`ModuleNotFoundError: No module named 'ultralytics'`（PC）

原因：没有在 `pc_train` 虚拟环境安装依赖。执行步骤 1 后重试。

`source: command not found`（PC）

原因：把 Linux 命令放到 Windows PowerShell 执行了。`source` 只能在板端 Bash 执行。

`No module named 'tbe'`（Board ATC）

先执行：`source /usr/local/Ascend/ascend-toolkit/set_env.sh && python3 -c "import tbe"`。若失败则检查 CANN 安装与 Python 环境。

`ais_bench: command not found`（Board）

可以使用 `python3 -m ais_bench`，或者直接用本项目 `run_infer.sh` + `infer_image.py` 路线。

输入密码后“没反应”

ATC 编译可能耗时几十秒到几分钟。开第二个 SSH 窗口检查：`ps -ef | grep atc | grep -v grep`。
